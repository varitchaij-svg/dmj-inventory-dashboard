# ข้อเสนอปรับปรุงระบบ — รอ Owner ตัดสินใจ

> เอกสารนี้รวมงานที่ "น่าทำแต่ยังไม่ทำ" จากการประเมินระบบรอบ ก.ค. 2026
> เรียงตามผลตอบแทน/ความพร้อมของข้อมูล — ทุกข้อยังไม่เริ่ม รอ owner ยืนยันก่อน

## 1. ตัวตนพนักงานรายบุคคล (ผลกว้างสุด แนะนำทำก่อน)

**ปัญหาตอนนี้:** ทุกการกระทำใน Audit Log บันทึกแค่ "role ไหนทำ" (เช่น `frontstore`)
ไม่ใช่ "ใครทำ" — ตอบคำถาม "ของหายใครเบิก / ใครนับผิด" ไม่ได้
และการสั่งของผ่าน LINE (`handleOrder_` ใน GAS) ไม่ลง audit log เลย

**ข้อเสนอ (ออกแบบให้เหมาะ low-tech):**
- หลังแตะเลือก role ในหน้า login → มีหน้าเลือก "ชื่อเล่น" อีก 1 แตะ
  (ปุ่มใหญ่ รูป+ชื่อ ไม่ต้องพิมพ์อะไร) → เก็บใน sessionStorage
- ทุกจุดที่ส่ง `actor` เปลี่ยนจาก role เป็น `role:ชื่อเล่น` (เช่น `frontstore:แนน`)
- แก้ `handleOrder_` ให้เรียก `writeAuditLog_` ด้วย
- รายชื่อพนักงานเก็บใน Google Sheet แผ่นใหม่ (owner แก้เองได้ ไม่ต้องแก้โค้ด)

**ต้นทุน:** UX เพิ่ม 1 แตะตอน login · งานแก้ ~1 วัน · เปิดทางทำ dashboard
สรุปงานรายคนต่อในอนาคต

## 2. กำไร/Margin ใน analytics — **แนะนำ "ไว้ก่อน" จนกว่า owner ยืนยันสูตร**

**โครงสร้างราคาที่เข้าใจตอนนี้** (จากพนักงาน — ยังไม่ได้ยืนยันกับ owner):
```
ราคาในระบบ = ราคาปลีก
ราคาส่ง    = ราคาปลีก × 0.8        (ลบ 20%)
ราคาส่ง    ≈ ราคาทุน × 3 แล้วปรับให้ลงท้ายหลักหน่วยเป็นเลข 8
∴ ราคาทุนประมาณ ≈ (ราคาปลีก × 0.8) ÷ 3
```

**เหตุผลที่ควรรอ:**
1. ตัวคูณ ×3 อาจไม่เท่ากันทุกหมวด/ทุก supplier — ถ้าคำนวณ margin จากสูตรเดา
   แล้วตัวเลขผิด จะพาตัดสินใจผิดทั้งกระดาน (อันตรายกว่าไม่มีตัวเลข)
2. ระบบมี **ราคาทุนจริง** อยู่แล้วบางส่วน: ชีต "รายการซื้อสินค้า" (sync จาก ZORT PO)
   มี `unitPrice` ต่อ SKU — แม่นกว่าสูตรเดา ควรใช้เป็นแหล่งหลัก

**ข้อเสนอเมื่อพร้อมทำ (hybrid):**
- ทุนจริงจาก purchases `unitPrice` (ล่าสุดต่อ SKU) เป็นหลัก
- SKU ที่ไม่มีข้อมูลซื้อ → fallback สูตร `(ปลีก×0.8)÷3` และติดป้าย "ประมาณการ" ชัดๆ
- คำถามที่ต้องถาม owner ก่อนเริ่ม: ตัวคูณ ×3 ใช้ทุกหมวดไหม? ราคาปลีกในระบบ
  อัปเดตตามป้ายจริงเสมอไหม? ต้องการเห็น margin เป็น % หรือบาท/ชิ้น?

  เป็นรหัส default เดิม ('DMJ') ซึ่งเดาง่าย

  แต่ถ้าทำข้อ 1 แล้วควรเปลี่ยนเป็นตรวจ PIN/session แทน)
